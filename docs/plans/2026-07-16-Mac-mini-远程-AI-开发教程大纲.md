# 把 Mac mini 变成随时在线的 AI 开发主机

> 副标题：出差不断线、多设备共用环境、远程编程与微信真机调试完整教程

## 一、教程定位

这是一篇准备发布到飞书的单篇长教程。文档前半部分帮助读者理解背景、架构与工具职责，后半部分提供可以按目录跳转的安装和操作步骤。

教程同时面向两类读者：

1. **非技术创作者、产品经理和个体开发者**：会使用电脑和 AI 对话，但不熟悉终端、SSH、Docker 等开发概念。
2. **AI 编程初学者**：已经接触过代码和 Git，希望搭建一套稳定、可持续的远程开发环境。

正文按零基础读者能够理解的方式编写。必要的技术原理放在提示框中，不在主线中展开同类产品横评或高级工程化内容。

## 二、真实背景与核心问题

教程从以下四个真实问题切入：

1. 以前使用笔记本开发，但经常出差；路上网络中断、电脑合盖或没电后，AI 与开发任务无法继续执行。
2. 笔记本性能有限，同时运行 AI 编程工具、开发服务和数据库时容易卡顿。
3. 使用多台电脑开发时，代码、依赖和数据库需要反复同步，容易出现环境不一致。
4. 除固定开发主机外，希望其他电脑和手机也能连接开发环境、运行服务并查看尚未正式部署的 Web 或小程序。

教程需要传达的核心观念是：

> 不是为了远程控制一台 Mac mini 而堆砌工具，而是把开发环境集中到一台持续在线的个人开发主机。其他设备只负责连接、操作和预览。

## 三、教程完成标准

读者完成主线后应当能够：

- 从外地使用 Windows 笔记本安全连接 Mac mini；
- 在笔记本上编辑 Mac mini 内的代码并运行命令；
- 使用远程桌面操作微信开发者工具等图形界面软件；
- 让开发服务和 AI 任务在断网、合盖后继续运行；
- 更换电脑后继续使用同一套代码、依赖和数据库；
- 在其他电脑或手机上访问 Mac mini 运行的 Web 服务；
- 通过笔记本转发端口，在手机上完成微信小程序真机预览；
- 理解最基本的远程访问安全与备份边界。

## 四、整套工作流

### 4.1 设备分工

```text
Mac mini：唯一开发主机
├─ 保存代码、依赖和本地数据库
├─ 运行 Web、API 与 Docker 服务
├─ 运行 Claude Code、Codex 与长期任务
└─ 运行微信开发者工具

Windows 笔记本：远程控制台
├─ 通过 Tailscale 找到 Mac mini
├─ 通过 VS Code Remote SSH 编辑代码、运行命令
├─ 通过 VS Code 转发开发端口
└─ 通过 ToDesk 或网易 UU 操作 Mac mini 桌面

手机：预览与检查终端
├─ 访问转发后的 Web 或 API
└─ 扫描微信开发者工具二维码进行真机调试
```

### 4.2 三条远程通道

| 通道 | 作用 | 使用工具 |
|---|---|---|
| 私网通道 | 让外地设备安全找到 Mac mini | Tailscale |
| 代码通道 | 编辑文件、运行命令、转发端口 | VS Code Remote SSH |
| 桌面通道 | 操作微信开发者工具等 GUI 软件 | ToDesk / 网易 UU |

### 4.3 微信小程序真机调试链路

```text
手机
→ 访问笔记本地址
→ 笔记本上的 VS Code 转发端口
→ Mac mini 上运行的本地 API

同时：
手机扫描 Mac mini 上微信开发者工具生成的二维码
→ 打开开发版本小程序
→ 调用经笔记本转发的本地 API
```

这一场景的重点不是“完全绕过微信预览流程”，而是“不必先把 API 和 Web 服务部署到正式服务器，就能完成外出环境下的真机调试”。

## 五、详细大纲

### 第一篇：先看懂这套工作流

#### 第 1 章　我为什么需要一台持续在线的开发主机

##### 1.1 笔记本开发遇到的四个问题

- 出差路上断网、合盖和没电导致任务中断；
- 性能有限，多项服务同时运行容易卡顿；
- 多台电脑之间同步代码、依赖和数据库很麻烦；
- 本地服务难以在其他电脑和手机上随时查看。

##### 1.2 从“随身开发”改成“固定主机计算”

- Mac mini 负责持续运行；
- 笔记本不再保存第二套开发环境；
- 手机和其他电脑作为入口，而不是新的开发主机。

##### 1.3 这套方案解决什么、不解决什么

- 解决环境统一、任务持续和多设备访问；
- 不等于云服务器，也不替代正式部署；
- 网络完全不可用时仍无法远程操作，但 Mac mini 上已经启动的任务可以继续执行。

#### 第 2 章　最终要搭成什么样

##### 2.1 一张图看懂设备之间的关系

- 展示 Mac mini、Windows 笔记本、手机和互联网之间的连接关系；
- 使用生活化比喻解释“主机、控制台和预览终端”。

##### 2.2 为什么代码和数据库只保留一份

- 避免多设备环境漂移；
- 减少依赖重复安装；
- 更换客户端时无需重新搭建项目。

##### 2.3 为什么远程开发需要三条通道

- 私网通道解决“如何找到”；
- 代码通道解决“如何开发”；
- 桌面通道解决“如何操作 GUI 软件”。

#### 第 4 章　你是否适合使用这套方案

##### 4.1 适合的人

- 经常出差或在多地点办公；
- 经常运行 AI 编程任务；
- 需要 macOS 或微信开发者工具；
- 希望电脑和手机随时访问开发环境。

##### 4.2 可能不需要完整搭建的人

- 只在一台电脑上偶尔修改简单项目；
- 不需要长期任务、远程预览或图形界面软件；
- 无法为固定主机提供稳定电源和网络。

##### 4.3 搭建前必须接受的现实

- 固定主机需要保持开机和联网；
- 远程入口必须做好账号与密钥保护；
- 远程主机仍然需要备份；
- 第一次搭建比普通本地开发更复杂，但之后更换设备会更轻松。

#### 第 5 章　搭建路线与验收标准

##### 5.1 第一阶段：能连接

- Tailscale 中能看到两台设备；
- Windows 能通过 SSH 登录 Mac mini；
- VS Code 能打开 Mac mini 上的项目目录。

##### 5.2 第二阶段：能持续工作

- 开发服务在 Mac mini 上运行；
- AI 和终端任务放入 tmux；
- 笔记本断网重连后能恢复原会话。

##### 5.3 第三阶段：能跨设备预览

- 笔记本能访问 Mac mini 上的 Web；
- 手机能访问经过笔记本转发的 API；
- 手机能扫描开发者工具二维码并完成小程序真机调试。

### 第三篇：日常怎么用

#### 第 17 章　出差时开始一天的开发

- 确认 Tailscale 在线；
- 使用 VS Code 连接 Mac mini；
- 恢复项目 tmux 会话；
- 检查开发服务；
- 需要 GUI 时再打开远程桌面。

#### 第 18 章　网络中断后如何继续任务

- 区分“连接中断”和“Mac mini 上的任务停止”；
- 重新连接 Tailscale 与 SSH；
- 恢复 VS Code 工作区和 tmux 会话；
- 检查 AI 任务与开发服务输出。

#### 第 19 章　如何在电脑之间切换而不搬开发环境

- 新电脑只安装连接工具；
- 不复制 `node_modules` 和本地数据库；
- 从同一台 Mac mini 继续项目；
- 正确同步 SSH 与账号设置。

#### 第 20 章　如何让 Claude Code 与 Codex 分工

- 用一个工具实现，用另一个工具审查；
- 先看修改计划，再允许执行；
- 检查 diff 和测试结果；
- 不让 AI 自动提交、推送或接触生产密钥。

#### 第 21 章　如何远程查看 Web 和小程序

- 查看 Web 时只需要代码通道和端口转发；
- 查看小程序时还需要桌面通道和开发者工具二维码；
- 用一张流程卡分别列出电脑预览与手机真机调试步骤。

#### 第 24 章　常见故障排查

按连接链路从外到内排查：

1. Mac mini 是否开机和联网；
2. Tailscale 是否在线；
3. SSH 是否能连接；
4. VS Code Remote SSH 是否正常；
5. tmux 中的任务是否还在；
6. 开发服务是否监听正确端口；
7. VS Code 端口是否正确转发；
8. 手机访问的是否为笔记本地址；
9. 微信开发者工具是否生成了正确的预览版本。

#### 第 25 章　其他设备的替代接入方式

本章不做产品横评，只说明设备变化后的对应关系：

- Windows 客户端换成 macOS 后，整体架构不变；
- 手机系统变化不影响 Web 访问的基本链路；
- 如果项目不需要 GUI 软件，可以不安装远程桌面；
- 如果项目不需要数据库，可以暂不安装 Docker；
- 如果只使用一种 AI 编程工具，可以省略另一种。
