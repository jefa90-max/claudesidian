# Gstack + Superpowers 完整开发流程最佳实践

> **核心理念：Think → Plan → Build → Review → Test → Ship → Reflect**
>
> 技能之间是链式调用的：每一步都知道前一步产出了什么，下一步需要什么。

## 各阶段详解

### Phase 0: 产品思考 — `/office-hours`

**何时用：** 需求不清晰、方向不确定、想验证想法时

这是 YC Office Hours 模式。它会用 6 个逼问问题拷打你的想法：
- 你到底在解决什么问题？
- 用户现在怎么解决这个问题？（现状方案）
- 为什么用户会迫切需要你的方案？
- 你有什么不公平的优势？

**产出：** 一份经过审视的需求描述或设计文档

---

### Phase 1: 头脑风暴 — `/brainstorming`

**铁律：方案未批准前，一行代码都不写。**

9 步流程：
1. **探索项目上下文** — 读文件、文档、最近提交
2. **提供可视化伙伴**（如有需要）
3. **逐个澄清问题** — 单选优先，一次只问一个
4. **提出 2-3 个方案** — 含权衡分析和推荐
5. **分段展示设计** — 每段获得用户批准后继续
6. **写设计文档** → `docs/superpowers/specs/YYYY-MM-DD-<主题>-design.md`
7. **自审** — 扫描占位符、矛盾、歧义
8. **等待用户审阅文件**
9. **自动调用** → `/writing-plans`

---

### Phase 2: 编写计划 — `/writing-plans`

**产出：** `docs/superpowers/plans/YYYY-MM-DD-<功能名>.md`

计划要求：
- 每步 **2-5 分钟**可完成
- **真 TDD**：写失败测试 → 确认失败 → 写最小实现 → 确认通过 → 提交
- **零占位符**：禁止 "TBD"、"TODO"、"添加适当的错误处理"、"类似任务N"
- **自审**：规范覆盖检查、占位符扫描、类型一致性检查

---

### Phase 3: 计划评审 — `/autoplan` (一键四审)

**何时用：** 计划已编写完成，想从多维度审视计划的可行性

| 审查角色 | 技能 | 关注点 |
|---------|------|--------|
| CEO/创始人 | `/plan-ceo-review` | 方向对不对？10 星产品是什么样？ |
| 设计师 | `/plan-design-review` | 设计维度 0-10 打分，低分项如何提升到 10 |
| 工程经理 | `/plan-eng-review` | 架构、数据流、图表、边界情况、测试方案 |
| DX 专家 | `/plan-devex-review` | 开发者人设、竞品对比、20-45 个逼问 |

也可单独调用某一个审查。`/autoplan` 会依次运行全部四个。评审发现问题后回到 Phase 2 修改计划，形成闭环。

---

### Phase 4: 隔离工作区 — `/using-git-worktrees`

**为什么需要：** 避免在主分支上直接开发，保持工作区干净

自动处理：
- 检测已有 worktree 目录或询问用户偏好
- 确认目录被 `.gitignore` 忽略
- 创建 worktree 并自动运行项目安装（npm install / pip install 等）
- 运行测试验证干净基线

---

### Phase 5: 执行实现

#### 推荐路径：`/subagent-driven-development`

每个任务的循环：
```
派发实现子代理（携带完整任务文本，不让子代理读计划文件）
    ↓
处理子代理状态：DONE / DONE_WITH_CONCERNS / NEEDS_CONTEXT / BLOCKED
    ↓
派发规范合规审查子代理 → 修复 → 重审直到 ✅
    ↓
派发代码质量审查子代理 → 修复 → 重审直到 ✅
    ↓
标记任务完成 → 进入下一个任务
```

**关键红线：**
- ❌ 不要并行派发多个实现子代理（会冲突）
- ✅ 规范审查必须在质量审查之前
- ✅ 提供完整任务文本，不要让子代理自己读文件

#### 备选路径：`/executing-plans`

当子代理不可用时，主会话逐步执行。遇到阻塞立即停下来问，不要猜。

### 实现中的三条铁律

#### 铁律 1: TDD — `/test-driven-development`

```
🔴 RED:    写一个最小的失败测试
           ↓  必须运行确认失败（且原因正确）
🟢 GREEN:  写最简代码使测试通过
           ↓  必须运行完整测试套件确认通过
🔵 REFACTOR: 消除重复，改善命名
           ↓  保持绿色，不加新行为
🔁 重复
```

**如果先写了代码再想补测试？** 删掉代码，从测试重新开始。不保留为"参考"。

#### 铁律 2: 系统化调试 — `/systematic-debugging`

```
Phase 1: 根因调查 → 完整读错误信息、复现、检查最近变更、追溯数据流
Phase 2: 模式分析 → 找到可工作的类似代码，对比每一处差异
Phase 3: 假设验证 → "我认为X是根因因为Y"，最小变更测试
Phase 4: 实现修复 → 先写失败测试（调用TDD），一次一个修复
```

**3 次修复失败后：** 停下来，质疑架构。每次修复暴露不同位置的新问题 = 架构问题。

#### 铁律 3: 完成前验证 — `/verification-before-completion`

**5 步门禁：**
1. 确定：什么命令能证明这个声称？
2. 运行：**全新执行**（不是上一次的结果）
3. 阅读：完整输出 + 退出码 + 失败计数
4. 验证：输出是否确认了声称？
5. 声称：只有此时才能说"完成了"

**禁止的说法：** "应该可以"、"大概通过了"、"看起来正确"

### Phase 6: 代码审查（两层审查机制）

代码审查分为**开发中轻审**和**发布前终审**两层，不是二选一：

```
Task 1 完成 → /requesting-code-review   ← 轻量快审，几十秒
Task 2 完成 → /requesting-code-review
Task 3 完成 → /requesting-code-review
...
全部完成   → /review                    ← 重量级终审，多专家并行
           → /ship
```

#### 第一层：开发中轻审 — `/requesting-code-review`

- **频率：** 每完成一个 task 后都跑
- **机制：** 派一个 code-reviewer 子代理，对比计划检查实现是否正确
- **耗时：** 快，几十秒
- **目的：** 防止问题在任务之间滚雪球

#### 第二层：发布前终审 — `/review`

- **频率：** 一个分支跑一次，发布前
- **机制：** 完整审查流水线
  - 范围漂移检测（做多了还是做少了？）
  - 计划完成审计（plan 里的每一项都实现了吗？）
  - 清单检查（SQL 安全、竞态、LLM 信任边界等）
  - 专家军团并行审查（测试 / 安全 / 性能 / 数据迁移 / API 契约 / 设计）
  - 对抗性审查（Claude 子代理 + Codex 交叉验证）
  - 红队审查（200+ 行 diff 自动触发）
  - Fix-First 自动修复（机械问题直接修，复杂问题问你）
- **耗时：** 几分钟
- **目的：** 全方位扫描，产出 PR 质量评分，结果会被 `/ship` 读取

#### 配套技能

| 技能 | 用途 |
|------|------|
| `/receiving-code-review` | 处理审查反馈：完整阅读 → 理解 → 验证 → 评估 → 回应 → 逐项实现 |
| `/codex` | 派 OpenAI Codex 做独立第二意见审查（`/review` 已内置，单独用于额外审查） |

**处理反馈的禁忌：** 不要说 "Great point!" — 直接修代码。

---

### Phase 7: 质量保障

| 技能 | 功能 | 是否改代码 |
|------|------|-----------|
| `/qa` | 系统 QA 测试 + 自动修 bug + 生成回归测试 | ✅ 改 |
| `/qa-only` | 仅出结构化报告（含健康分数） | ❌ 不改 |
| `/design-review` | 设计师视角 QA + 6 项可用性测试 + 前后截图 | ✅ 改 |
| `/cso` | 安全审计：密钥考古、依赖供应链、CI/CD | ❌ 报告 |
| `/health` | 代码健康仪表盘（类型检查、lint、测试、死代码） | ❌ 报告 |
| `/benchmark` | 性能基线 + Core Web Vitals 对比 | ❌ 报告 |

**设计审查的 6 项可用性测试（v0.17.0 新增）：**
1. 主干测试 — 能看出这是什么网站、什么页面吗？
2. 3 秒扫描 — 用户首先看到什么？
3. 页面区域测试 — 能说出每个区域的用途吗？
4. 废话检测 — 多少内容是"废话"？
5. 无脑选择审计 — 每次点击都很明显吗？
6. 好感度储备追踪 — 什么在消耗用户的耐心？

### Phase 8: 发布上线

```
/ship                →  合并主分支 → 跑测试 → 审计覆盖率 → bump版本
                        → 更新CHANGELOG → 提交 → 推送 → 开PR
/finishing-a-development-branch  →  4个选项：
                        合并回主分支 | 推送开PR | 保留分支 | 丢弃
/land-and-deploy     →  合并PR → 等CI通过 → 等部署 → 验证生产环境
/canary              →  上线后持续监控：控制台错误、性能退化、页面故障
/document-release    →  读所有项目文档 → 对比diff → 更新README/ARCHITECTURE等
```

## 依赖链图

```
/office-hours (可选，产品思考)
  └─→ /brainstorming (必选，方案设计，产出设计文档)
        └─→ /writing-plans (必选，设计文档 → 分步执行计划)
              └─→ /autoplan 或单独审查 (可选但推荐，四维评审计划)

                    └─→ /subagent-driven-development (推荐)
```
