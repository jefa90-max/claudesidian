## 05\. 执行能力：AI Agent 智能体

💡

执行能力：让 AI 给你干活，用 AI Agent 实现自动化

本章最重要有四件事情：

- 通过学习，知道 Agent 是什么；

- 能够理解搭建 Agent 的底层技能是什么；

- 能够通过日报 Agent 这个案例，掌握底层技能的运用；

- 进阶：能够对技能进行迁移，比方说根据自己的场景需求去搭建 Agent（作业内容）

⚠️ 注意事项：

- 在学习过程中，大家一定要培养起万事问 AI 的习惯，有任何不清晰或者是想法都可以直接问 AI

- 本章最重要的不是案例，案例只是为了让大家感受到底层技能的运用，掌握技能后，就可以迁移到其他场景（案例有国内工具 Coze 版和国外工具 Codex 版二选一）；

- 为了更好的掌握底层技能的运用，大家一定要完成一份作业，只有在实操中才能熟练运用

1\. 理解执行能力与 AI Agent 智能体

![](https://sphere-sh.oss-cn-shanghai.aliyuncs.com/private/docx/drive/media/AoydbeZU5owU8Sxfc33cCPNTnGc_hhkc?x-oss-credential=LTAI5tDANFnEjK2qWnHwZKzr%2F20260508%2Fcn-shanghai%2Foss%2Faliyun_v4_request&x-oss-date=20260508T000447Z&x-oss-expires=86400&x-oss-signature=dbc6a1667d82f6f1baaf84ffc595fef2aa2e89470455ff8c06a0306e18499491&x-oss-signature-version=OSS4-HMAC-SHA256)

在前面几个章节中，我们学习了思考、积累、表达和产品能力，也开始搭建知识库、整理素材、生成内容，甚至做出个人信息网站。

但这些事情，本质上仍然是你在执行。

AI 只是你的工具——你主动去提问了它才给你回复。本质还是需要人找 AI 去聊，然后生成内容，手动复制结果，去进行下一步的任务处理。

这就是对话式 AI 用法的天花板：AI 能帮你，但不能替你干活。

而执行式 AI （Agent）就弥补了这个缺点，Agent = 你的 AI 员工，能自动化替你干活

这也是 Agent 为什么强， 因为它真正改变了你的工作方式：

- 以前：你跟 AI 聊天 → AI 给结果 → 你自己执行。你是主力，AI 是助手。

- 现在：你告诉 Agent 要做什么 → Agent 自己读信息、自己处理、自己输出。Agent 是主力，你只看结果。

价值点一： Agent 可以帮你干活，并且速度更快，效果更好

Agent 可以连接微信、飞书、文档、网页、知识库等工具，自己去不同平台获取信息、处理内容，并输出到指定位置。你从“每一步都亲自做”，变成“只看最终结果”。

价值点二：把你从繁琐事项中解放出来，且自动化执行

Agent 把流程定一次，它就自己从头跑到尾——上一步的输出自动接到下一步，中间不用你点"继续"；多条任务还能同时开工，2 小时手动的活 10 分钟跑完。

且一次设定，反复执行。你花一次精力把流程搭好，以后每天/每周它自动跑，你不用重复劳动。

价值点三：稳定、可并行干活，相当于招聘多个员工

人只能高效高质量去做一件事的——做完 A 才能做 B。而且事情做久了之后注意力会直线下降。

但是 Agent 可以同时跑多个任务，而且每次输出质量都很高。

比如做小红书自媒体，一个人也可以借助 Agent 跑出一条小团队流水线：

- 二创 Agent：把选好的话题，按你的语气改写成标题、封面文案和正文

- 发布 Agent：同步发布到小红书、公众号、视频号，并适配不同平台格式

- 复盘 Agent：汇总点赞、评论、涨粉数据，判断哪些选题值得继续做

你的角色从"一个人从头干到尾"变成"挑选题、定风格、看数据做判断"。

选题到发布到复盘，全自动流水线——这就是 OPC（一人公司）能跑起来的底层逻辑。

2\. 搭 Agent 的核心技能：和 AI 聊清楚需求

![](https://sphere-sh.oss-cn-shanghai.aliyuncs.com/private/docx/drive/media/YEyob4GnGotyygxJMbDcfBl9n3g_hhkc?x-oss-credential=LTAI5tDANFnEjK2qWnHwZKzr%2F20260507%2Fcn-shanghai%2Foss%2Faliyun_v4_request&x-oss-date=20260507T180508Z&x-oss-expires=86400&x-oss-signature=d3688b1ade6c6cc70c0a7d50065b8584ccd61817fdd5794ee70d31f8dfd98099&x-oss-signature-version=OSS4-HMAC-SHA256)

在上一步介绍 Agent 的时候会发现调用了很多能力，比如内容生成，联动飞书，发布到平台。

乍一看就是一个很复杂的系统，还要处理逻辑全部想清楚。还得会写 prompt、调参数……光是看到这些，就已经劝退 80% 的人了。

但真相是：这些事，AI 都能帮你做。 AI 已经进化到自己写软件，自己操作网页，文件了

技术活 AI 全包了。你唯一要做的，是想清楚你要什么，然后用自然语言说出来就行

所以，搭 Agent 就剩下一个最核心能力：跟 AI 把需求聊清楚。

你不需要掌握这些背后的技术，只需要跟它一起拆解清楚你要做的是什么。需求拆清楚了，输入输出自然就明确了，流程自然就能固化下来了。

2.1 为什么你搭建 Agent 不成功

![](https://sphere-sh.oss-cn-shanghai.aliyuncs.com/private/docx/drive/media/EGnNbh6Tmo4ly3xXQ4ocCMdQnYb_hhkc?x-oss-credential=LTAI5tDANFnEjK2qWnHwZKzr%2F20260508%2Fcn-shanghai%2Foss%2Faliyun_v4_request&x-oss-date=20260508T000447Z&x-oss-expires=86400&x-oss-signature=2beeff744caf2875e424c385b086dfa49ef5b057a8ffa9677ef601ff447888e8&x-oss-signature-version=OSS4-HMAC-SHA256)

核心结论：大多数人失败，不是工具问题，而是需求本身没想清楚。

这里需要一个关键的认知转变：你是老板，把 Agent 当成你的数字员工，而 AI 是帮你培训这位员工的角色。

老板的核心能力是什么？不是自己会写代码，而是能把需求交代清楚。

- 模糊指令的示范： 你告诉 AI：“帮我整理一下资料。” AI 基于这句模糊的话，给出一堆方案。结果往往不是你想要的。不是 AI 不行，是你给的需求不清晰。

- 清晰指令的示范：

a. “我每天收藏公众号文章，但从不回头看。”

b. “请每周帮我整理一次，提取核心要点，标出值得复习的部分。”

c. “结果按时间顺序，发布到我的飞书文档。”

AI 立刻就能给出精准、可执行的方案。需求越清晰，结果越可控。

所以，搭 Agent 的核心不是学技术，而是学会当一个“好老板”——你不需要比 AI 更懂技术，只需要比它更懂“自己要解决什么问题”。

那是不是自己把需求想清楚就够了？还不够。因为个人思考存在“盲区”——你意识不到自己“不知道什么”。

比方说，你是一个小红书博主，最近涨粉变慢，想搭一个 Agent 帮你优化选选题。

而真实问题，涨粉慢可能不是因为热点追得少，而是追了太多与账号定位无关的热点，导致内容混乱，粉丝记不住你。

那如果你不是直接告诉 AI "我要追热点"，而是让 AI 先问你几个问题呢？

比如：

- “你最近涨粉慢，是新关注少了还是取关多了？”

- “你过去数据最好的三篇内容是什么类型？”

- “你现在的选题来源是什么？”

几个问题下来，AI 可能发现：你数据最好的内容是深度干货，但最近却在追娱乐热点，导致内容与粉丝预期严重不符。真正的问题不是“热点追得不够快”，而是“选题偏离了账号定位”。

这就是盲区：你自己想不到，直接问 AI 也问不到。因为 AI 只会顺着你给定的方向回答。

你问 AI，只能问到你已经想到的。你的盲区，永远是盲区。

所以需求不是想出来的，是被问出来的——要反过来，让 AI 来问你。

![](https://sphere-sh.oss-cn-shanghai.aliyuncs.com/private/docx/drive/media/PbySbOPVUovC5ax4jJLc8sENnuc_hhkc?x-oss-credential=LTAI5tDANFnEjK2qWnHwZKzr%2F20260507%2Fcn-shanghai%2Foss%2Faliyun_v4_request&x-oss-date=20260507T180508Z&x-oss-expires=86400&x-oss-signature=0506c9c36cd20747b3dffe0109e75d66c6f6f622ac8a4692ae1fe5b24ee72bd2&x-oss-signature-version=OSS4-HMAC-SHA256)

2.2 怎么做：把 AI 切换需求顾问模式

![](https://sphere-sh.oss-cn-shanghai.aliyuncs.com/private/docx/drive/media/NnYpbdsEroVt4BxKNrdcWpyxnZe_hhkc?x-oss-credential=LTAI5tDANFnEjK2qWnHwZKzr%2F20260507%2Fcn-shanghai%2Foss%2Faliyun_v4_request&x-oss-date=20260507T180508Z&x-oss-expires=86400&x-oss-signature=fc700026f5c710ce9ae3beda7179ec96e9f4ecce5dc26030b89093fdd2ebdbdb&x-oss-signature-version=OSS4-HMAC-SHA256)

明白了盲区问题，解决方法就一句话：反转提问权。 不要让 AI 直接出方案，而是让它来追问你，帮你理清需求。

![](https://sphere-sh.oss-cn-shanghai.aliyuncs.com/private/docx/drive/media/P1JLbLPTWo2rJ9xh0OYcMUMmn0f_hhkc?x-oss-credential=LTAI5tDANFnEjK2qWnHwZKzr%2F20260508%2Fcn-shanghai%2Foss%2Faliyun_v4_request&x-oss-date=20260508T000447Z&x-oss-expires=86400&x-oss-signature=4799f1ca6613337e56ce96ead562628f996367e5cf0c839d7c95ce8d1b59b85a&x-oss-signature-version=OSS4-HMAC-SHA256)

具体操作（可直接复制）：

第一步：想一个你打算用 Agent 解决的简单问题。

第二步：对 AI 使用下面这段提示词（把 xxx 换成你自己想做的事）：

```
我想做一个 xxx Agent，但现在这个想法还比较模糊。
请你不要直接给我完整方案，也不要先推荐很多平台。

请你先像一个智能体需求顾问一样，帮我分析这个想法。
你现在的任务不是给方案，而是辅助我做需求判断和决策分析。
请根据我的回答，逐步追问那些最关键的不确定点。

要求：
1. 每轮最多问我 3 个问题
2. 不要一次性展开很多功能
3. 优先围绕"解决什么问题、哪一段最值得先做、需要什么能力、结果落到哪里"来追问
4. 最后再帮我收束成一个最小可用 demo
```

这段提示词可以改，甚至可以不用，用你自己的话说也行。

你不需要自己对着这张清单逐条填空——进入顾问模式后，AI 会顺着你的回答把这些一个个问出来

为什么必须这么做？

因为 AI 的默认模式是“你问它答”。你问的方向，就决定了 AI 输出的边界。 如果你的方向本身是错的（比如“怎么追热点”），AI 会在错误框架内，给你一个看似完美、实则无用的方案。

而反转提问权，就反转了信息优势，原因如下

- 你是新手，AI 是“老手”： 你是第一次提某个需求，但 AI（基于其训练数据）已经见过无数类似案例。它知道常见的坑和关键决策点。

- 经验只在“提问”时激活： AI 的这些经验，在“你问它答”模式下是沉默的。只有当你让它来提问时，它才会基于经验，去追问那些你很可能忽略的关键点（例如：“你数据最好的内容类型是什么？”).

- 追问发现更多的价值： 每一个问题都是一个“认知探针”，帮你把脑子里已有、但从未结构化表达的想法“抽”出来。很多需求不是你“不知道”，而是你“没想到要说”。比如，AI 问你“灵感从哪来？”，你才意识到“我都是刷笔记时随手记的”——这直接决定了 Agent 的输入源。

第三步：和 AI 聊天，在 AI 的追问中梳理 Agent 的核心纬度

AI 开始提问后，你只需要诚实回答，因为 AI 追问的方向，就是搭 Agent 必须搞清楚的五个维度。你会在回答中，不知不觉把这些问题全部定义清楚：

1. 核心痛点： 到底要解决什么问题？（避免做无用功）

2. 最小范围： 当前最值得先做哪一段？（确保能完成）

3. 输入数据： 需要什么信息，从哪来？（确保 Agent 有粮可加工）

4. 处理流程： 中间需要哪些步骤？（确保逻辑通畅）

5. 输出位置： 结果落到哪里，长什么样？（确保结果可用）

所以，和 AI 聊需求的过程，本身就是设计 Agent 方案的过程。 而拆解需求这一个动作，就把搭 Agent 需要的所有关键信息都问出来了。

输入输出、处理流程、输出位置——这些不是你另外要掌握的能力，是在和 AI 交流、拆解需求的过程中自然产生的。需求拆得够清楚，这些东西就全都有了。

到这一步，很多新手可能会困惑：“聊完了，然后呢？这些技术细节我不会啊。”

请记住这个核心认知：你不需要会技术。

- 你的角色是“产品经理/老板”，你的核心产出是一份清晰的需求定义（即上面五个维度）。

- AI 的角色是“技术专家/员工”，它的职责是把你的需求定义，翻译成可执行的技术方案，生成 Agent。

当聊得差不多时，你只需要让 AI 给你收束成最小方案，然后让 AI 去开发 Agent 即可。

因为 AI 会基于你们完整的聊天记录，理解你的全部意图，直接给出包含具体工具、步骤、配置方法的方案。那些“输入输出”、“处理流程”的技术实现，是 AI 基于你的需求描述自动完成的，不是你额外要学的。

总结一下：AI 开启提问模式（确定需求）——梳理核心纬度——产出 Demo—— AI 生成 Agent

第四步：收束成最小 Demo

聊完后，你可能会得到一堆功能点。但你不能全做——功能越多，Agent 越容易卡住。

必须收束： 把所有聊过的功能砍到最小，只保留最核心的一条链路，先把它跑通。

这就是 “最小 Demo” ——不是最终产品，而是一个能验证想法、可以跑起来的最简版本。就像盖房，先搭一间能住的小屋，确认地基稳了，再往上加。

具体操作：当需求聊得差不多时，对 AI 发出明确的收束指令：

```
请根据刚才我们的对话，不要继续发散，直接把结论收束成一个最小可用 demo。

请明确写出：
1. 当前先做哪一段
2. 这段要解决的核心问题是什么
3. 需要什么能力
4. 输入是什么
5. 输出是什么
6. 结果先落到哪个位置
7. 当前先不做什么
```

从一句模糊的话，变成了一份可以直接动手做的设计图纸。 后面固化成 skill、验证迭代，都基于这份图纸来做。

到这一步，你做的就是思考能力出现的 Harness 思路——先让 AI 帮你发散，再把发散的结果收束成可执行的东西。

先让 AI 把模糊想法追问透，再收束成一份最小可用 Demo 的图纸 ，发散是为了打破盲区，收束是为了能落地，这就是 Harness 思路在 Agent 搭建中的应用

第五步：生成 Agent

把 Demo 交给 AI（如 Codex / Coze）去搭建，这是最关键的认知改变：你不需要自己动手写一行代码去“搭建”。

所以，拿到 Demo 后，你只需要：

1. 打开你的 AI 编程工具（如 Codex、Coze、Claude 等）。

2. 把这份“最小 Demo 定义”文档丢给它。

3. 给出明确的指令：“请根据这份需求文档，帮我搭建/实现这个 Agent。请生成完整的代码/配置/工作流。”

AI 编程工具会理解你的需求，自动生成代码、配置技能（Skill）、串联工作流，把这个 Agent 给你“搭”出来。 你可能会需要与它进行几轮调试和微调，但核心的生产力是它，不是你。

2.3 Agent 和 Skill 的关系（拓展内容）

在后面的章节里，你会看到两个词：Skill 和 Agent。它们很像，但不是一回事。先说清楚，后面就不会绕。

Skill = 一个单一能力

比如"把今天的日报总结成结构化要点"——这就是一个 Skill。

它做一件事，做得稳定，可以被反复调用。你可以把 Skill 理解为一个"专项技能"：分析日报是一个 Skill，提取小红书爆款标题是一个 Skill，把网页正文转成 Markdown 也是一个 Skill。

Agent = 把多个 Skill 串起来自动跑的"员工"

比如：每天定时读取 日报。txt → 调用「日报分析」Skill → 把结果写到汇总文档 → 同步到飞书。

整条链路不用你点"继续"，它自己从头跑到尾。

在接下来的日报案例中，两者大约有 90% 是重合的——一个 Skill 自己也能"分析总结"。

Agent 多出来的那 10%，恰恰是关键的部分：

• 自动输入：自己去读文件 / 抓网页 / 拉数据，不用你手动喂

• 自动触发：定时跑、监听变化跑，不用你每次点开

• 多步联动：上一步的结果自动接到下一步，跨工具跨平台串成一条流水线

先有 Skill，再有 Agent，这是搭智能体最自然的顺序。

请你先像一个智能体需求顾问一样，帮我分析这个想法。

你现在的任务不是给方案，而是辅助我做需求判断和决策分析。
请根据我的回答，逐步追问那些最关键的不确定点。

要求：
1. 每轮最多问我 3 个问题
2. 不要一次性展开很多功能
3. 优先围绕“解决什么问题、哪一段最值得先做、需要什么能力、结果落到哪里”来追问
4. 最后再帮我收束成一个最小可用 demo
```

接下来，重点不是一次性把所有想法都说完，而是顺着 Codex 的问题往下答。这个环节非常重要，且需要你耗费较多时间，通过和 AI 交流，聊清楚你的需求，让 AI 知道你要什么

这一轮交流先跟着问题走，聊到这里，你会慢慢发现：你会慢慢把当前最核心的一些需求功能点梳理出来

3.2.3 第三步：让 Codex 收束最小 demo

不需要所有问题都回答，当问题问得差不多以后，你觉得基本的信息都告诉 Codex 之后，让 Codex 直接收结论。

```

请明确写出：
1. 当前先做哪一段
2. 这段要解决的核心问题是什么
3. 需要什么能力
4. 输入是什么
5. 输出是什么
6. 结果先落到哪个位置
7. 当前先不做什么

3.2.4 第四步：让 Codex 整理成日报 Skill

前面这条流程收清楚之后，就可以让 Codex 整理成 skill 了，因为到这时候，skill 承接的已经不是“一个模糊需求”，而是一条相对稳定的固定动作。

```
请根据刚才我们已经收束好的日报 Agent demo，
不要重新发散需求，
直接把这套流程整理成一个可复用的日报 skill。

要求：
1. 先明确这个 skill 的职责边界
2. 写清楚输入、处理步骤、输出格式、最终检查项
3. 输出一个适合做成 SKILL.md 的版本
```

3.2.5 第五步：验证流程是不是通的

skill 整理完以后，不要马上继续扩展。先拿一段最小输入，让 Codex 按这个 skill 跑一轮。

```
今天做了 3 件事：
1. 看了 8 个对标账号，拆了 3 篇标题
2. 改了一版虚拟资料详情页
3. 梳理了日报 Agent 的需求

今天最大的卡点：
详情页卖点还是不够清楚，不知道该先强调结果还是先强调适合谁。

明天准备做：
1. 再拆 5 个同行详情页
2. 把日报 Agent demo 跑一遍
```

3.2.6 最终日报 Agent（进阶可选）

到上一步为止，你做的其实是一个「日报 Skill」——你发内容，它给你结构化分析。但每次都要你自己去复制结果，这一段重复劳动还在你身上。

对应前面我们说过的：Skill 是"能做这件事"，Agent 是"自己把这件事每天都做完"。

而把"自动存储"功能接到分析后面，链路就从「Skill」升级成了一个最小的「Agent」：你发内容 → 自动分析 → 自动写到固定位置 → 给你一句反馈。

下面就让 Codex 把"自动存储"加到 Skill 后面。存到哪里没有标准答案，挑你日常一定会打开的地方：

- 桌面 日报。txt：最简单的兜底方案，先跑起来再说

- 飞书文档 / Notion：适合记录笔记、复盘多的人

- Obsidian：适合已经在用本地知识库的人，可以直接和你的笔记网络联动

- 桌面 。md 文件 + 本地 html 网页：适合纯本地、不想登录任何平台的人（下面示例参考）

下面以"本地 html 网页"为例演示一种做法，其它形式同理，把目标存储位置告诉 Codex 即可。

```
我想要把日报存储成 md 格式，然后用网页查看历史的所有日报，
并把这个功能加进去 skill 中，
你帮我设计一个网页，纯本地双击打开 html 后可以加载查看 md 文件，
```

回头看一下这条链路是怎么搭起来的：

1. 拆需求：让 AI 追问你，把模糊想法聊清楚

2. 收束 Demo：把发散的对话钉成一份图纸

3. 固化 Skill：把分析能力沉淀成可复用的技能

4. 接上自动存储：从 Skill 升级成 Agent

跑通之后，也鼓励大家按自己的需要往下扩展。每个人的日报需求、工作流、展现偏好都不一样，这个 Agent 应该长成你的样子，而不是手册的样子。

下面是一些可以试的方向：

- 换存储形式：从 html 换到飞书 / Obsidian ，看看哪种你最方便

- 换分析口径：让它额外标出"今天的卡点""明天的优先级""本周累计推进"

- 接上别的输入源：比如自动读取微信、会议录音转写

- 串成更长链路：日报 → 周报 → 月度复盘，让它替你做汇总

执行能力=把重复工作交给 AI，一个人活成一支队伍

核心就四个动作：启动提问 → 梳理维度 → 收束成 Demo → 生成 Agent。

对应最重要的底层技能——需求定义能力（核心中的核心）

搭 Agent 不是技术活。 你的核心价值不是写代码，而是通过“反转提问权”，让 AI 当顾问来追问你，把“到底要解决什么问题、先做哪一段、输入输出是什么”这五个维度聊清楚。方向错了，AI 写得再快也是无用功。

恭喜你！到这里，不仅仅是做出一个日报 Agent，更重要的是，你掌握了如何通过和 AI 聊需求去做出 Agent，后续可以根据自己的需求去开发

掌握这个底层技能后，第三章的“表达能力”（如内容生成）和第四章的“积累能力”（如知识库），都可以被你封装成专属的 Skill，进而组合成更强大的 Agent，让你从内容的“生产者”升级为工作流的“架构师”。
