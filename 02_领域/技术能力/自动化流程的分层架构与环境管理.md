# 自动化流程的分层架构与环境管理

## 背景

在搭建"自动录屏 + 自动生成小红书内容"（xhs-reellingo-pipeline）和 ai-content-radar 等自动化 skill 的过程中，踩了不少坑，也总结出一些可复用的经验。

## 核心内容

### 1. 环境正确比"看起来差不多的参数"更重要

录屏链路调试中，确认了走 CDP Proxy（端口 3456）+ `bw-chrome` 容器的方式，比直接用 BrowserWing 8080 更稳定。**表面上参数差不多，但底层走的链路完全不同，最终录制结果差异很大。**

**经验：** 遇到"有时候行有时候不行"的问题，先怀疑环境是否一致，不要只看参数。

### 2. 自动化流程要分层

把一条自动化链路拆成以下几层，后续维护和接入 cron 都会轻松很多：

| 层 | 职责 | 示例 |
|---|---|---|
| env 配置 | 环境变量、密钥、端口 | `.env` 文件 |
| 前置检查 | 运行前自动检查/拉起依赖 | 检查 `bw-chrome` 容器、`cdp-proxy` |
| 运行入口 | 一键启动脚本 | `run_reellingo_once.sh` |
| skill | 可复用的流程编排 | `xhs-reellingo-pipeline` |
| schema | 输出结构定义 | `post_plan.json` schema |

### 3. 开发-测试-审查放在一个闭环

复杂项目推进时，不要把"开发完了等人测"和"测完了再审查"分成三段。放在一个闭环里完成，效率更高，上下文也不会断。

## 为什么重要

这套分层思路不只适用于小红书 pipeline，后续所有需要定时运行、持续维护的自动化任务都可以复用。**分层越清晰，出问题时定位越快，新功能接入越容易。**
